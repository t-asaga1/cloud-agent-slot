# パチスロアプリ 開発計画(提案)

作成: AGENT #001(プロジェクトキックオフ)

## 1. 提案の要点

- **技術スタック**: TypeScript + React + Vite(ブラウザで動く Web アプリ)
- **配布ターゲット**: Web アプリを主軸としつつ、同一コードベースから **Windows 向け exe(デスクトップアプリ)** も配布する(ユーザー承認済み方針)
- **設計方針**: ゲームロジックと UI を完全分離し、ロジックは純粋な TypeScript モジュールとして単体テスト可能にする
- **進め方**: 下記のフェーズを 1 フェーズずつ(場合によりさらに細かく)AGENT に割り当て、毎回動く状態でコミットする

## 2. 技術スタックの理由

| 項目 | 選定 | 理由 |
|---|---|---|
| 言語 | TypeScript | 型があるため引継ぎ主体の開発でも仕様が壊れにくい |
| UI | React + Vite | 起動・ビルドが速く、Cloud Agent 環境でブラウザ検証しやすい |
| テスト | Vitest | Vite と統合されており設定が最小で済む |
| 状態管理 | まずは React 標準(useReducer 等) | 規模が見えるまで外部ライブラリを増やさない |
| デスクトップ化 | Tauri(第一候補) / Electron(代替) | Web の成果物をそのままラップして exe 化できる。Tauri は成果物が軽量(数 MB)で Rust 側にロジックを書く必要はない |

ネイティブアプリ(iOS/Android)が最終目標の場合でも、ロジックを UI 非依存の TS モジュールにしておけば移植コストを最小化できる。

### Web + exe 両対応の方針

- アプリ本体はあくまで「Vite でビルドされる Web アプリ」。exe はそれをネイティブウィンドウでラップしたもの。
- したがって **通常の開発・テスト・レビューはすべて Web 側(ブラウザ)で行い、exe は配布用パッケージング工程としてのみ扱う**。
- ブラウザ API に依存する箇所(音声再生、永続化など)は薄い抽象を介して使い、デスクトップ側で差し替えが必要になっても局所化できるようにする。
  - 永続化: `localStorage` 直呼びではなく `src/platform/storage.ts` のようなラッパー経由にする
  - 音声: Web Audio API を `src/platform/audio.ts` 経由で使う
- exe のビルドは GitHub Actions(Windows ランナー)で自動化し、成果物を Release に添付する。Cloud Agent 環境(Linux)では exe の動作確認ができないため、CI ビルドの成功をもって検証とする。
- Tauri 導入は Phase 4(UI 最小実装)完了後の「Phase 4.5」として行う。それまでは Web のみで開発を進める。

## 3. アーキテクチャ方針

```
src/
  core/        # ゲームロジック(UI非依存・全て単体テスト対象)
    rng.ts         # 乱数(シード指定可能にしてテスト再現性を確保)
    lottery.ts     # 役抽選(設定別確率テーブル)
    reel.ts        # リール配列・停止制御(スベリ/引き込み)
    payout.ts      # 払い出し計算
    state.ts       # 内部抽選状態(通常3/CZ2/前兆1/ボーナス2/AT4)・メダル管理
  presentation/ # 演出層ロジック(ステージ移行抽選、演出選択。純関数でテスト対象)
  ui/          # React コンポーネント(リール表示、ボタン、動画背景・演出再生)
  platform/    # ブラウザ/デスクトップ差異を吸収する薄いラッパー(storage, audio, video)
  assets/      # 画像・動画・サウンド(管理ルールは docs/ASSET_GUIDELINES.md 参照)
src-tauri/     # (Phase 4.5 以降)Tauri の設定・exe ビルド用。ゲームコードは置かない
docs/
  SPEC.md      # 機種仕様書(役構成・確率・リール配列などの一次情報)
  ASSET_GUIDELINES.md  # 素材(画像・音声・液晶データ)の管理ルール
  HANDOVER.md  # 最新の引継ぎ資料
  handover/    # 引継ぎ履歴
```

## 4. フェーズ計画

各フェーズは「1 AGENT = 1 やり取り」の粒度を意識して分割している。

### Phase 0: 仕様策定(骨子作成済み・詳細は継続)
- `docs/SPEC.md` を作成する(AGENT #003 が骨子を作成済み。TBD 箇所の確定を継続する)
- 決定済みの前提(ユーザー指示):
  - リール本数 3 本・**各リール 20 コマ**
  - 内部抽選状態: 通常 3 種 / チャンスゾーン 2 種 / 前兆 1 種 / ボーナス 2 種 / AT 4 種(計 12 種)。レア役・ゲーム数で遷移
  - 演出(背景・BGM)も同数の状態(ステージ)を持つが、内部状態と 1:1 対応ではない(別レイヤーで管理し、ステージ移行抽選で対応付ける)
  - 背景・演出は基本的に動画(ループ WebM)で保持
- 残り決めること:
  - 機種コンセプト(オリジナル機。実在機種の仕様コピーは避ける)
  - 図柄の種類・リール配列(20 コマ)
  - 役構成と設定 1〜6 の役別当選確率テーブル
  - ボーナス/AT の仕様(純増・継続条件)・状態遷移テーブルの具体値
  - メダル仕様(投入3枚、払い出し枚数)

### Phase 1: コアロジック実装
- プロジェクト雛形作成(Vite + React + TS + Vitest)
- `core/` の rng・lottery・payout を実装し単体テストを書く

### Phase 2: リール制御

進め方の詳細(20 コマ前提):

1. **データモデル**: リールは「20 要素の図柄 ID 配列 × 3 本」。リール位置は 0〜19 のコマ番号で表し、表示窓(上中下段)は位置からの相対で導出する。
2. **停止制御の方式**: 実機同様の「テーブル方式」ではなく、**内部当選役からの引き込み優先度による探索方式**を採用する。
   - 停止ボタン押下時のコマ位置から最大 4 コマ先まで(計 5 停止候補)を探索し、当選役の構成図柄を有効ライン上に引き込める位置のうち優先度最高のものに停止する。
   - 優先度は「当選役を揃える > 当選役の一部を引き込む > ハズレ目(非当選役を蹴飛ばす)」。非当選役が揃う位置は候補から除外する(蹴飛ばし)。
   - この方式はコードとして読みやすく、リール配列変更(図柄差し替え)にもテーブル再作成なしで追従できる。
3. **押し順対応**: 第 1〜第 3 停止の順序と押下位置を入力に取り、押し順役(AT のナビ)に対応できるインターフェースにする。
4. **テスト**: 「当選役 × 全 20 押下位置 × 押し順」の全組み合わせで、(a) 当選役は 4 コマ以内なら必ず揃う、(b) 非当選役は絶対に揃わない、を網羅検証する単体テストを書く。100% 引き込み役(ベル・リプレイ)は配列側の設計(4 コマ間隔以内配置)もテストで保証する。
5. **回転アニメーションとの分離**: 回転・滑りの見た目は UI 層の責務とし、core は「押下位置 → 停止位置」の純関数のみ持つ(テスト容易性のため)。

### Phase 3: 遊技状態管理
- 内部抽選状態 12 種(通常3/CZ2/前兆1/ボーナス2/AT4)の遷移をデータ駆動テーブルで実装、メダル増減、ゲーム数管理
- レア役契機・規定ゲーム数契機の遷移と、1ゲームの流れ(BET → レバー → 抽選 → 停止 → 払い出し)を通しで動かすテスト

### Phase 4: UI 実装(最小)
- リール表示・レバー/停止ボタン・メダル枚数表示
- ブラウザで 1 ゲーム回せる状態にする

### Phase 4.5: exe パッケージング(Tauri 導入)
- Tauri を導入し、Web ビルドをそのまま Windows exe としてパッケージングできるようにする
- GitHub Actions(Windows ランナー)で exe ビルドを自動化し、Release に成果物を添付する
- `npm run build`(Web)と `npm run tauri build`(exe)の両方が通ることを CI で保証する

### Phase 5: 演出・仕上げ
- 演出ステージ管理(内部状態とは別レイヤー。ステージ移行抽選は `src/presentation/` の純関数として実装しテスト)
- 状態別ループ背景動画(WebM)+ BGM の切り替え再生、演出動画(カットイン等)のトリガー再生
- 停止時の演出、告知、サウンド(SE)
- 履歴表示(ゲーム数、ボーナス回数、スランプグラフ)
- 設定変更機能、オートプレイ/シミュレーションモード

※ ユーザー素材(筐体画像・背景動画・図柄・BGM・SE)の入稿と取り込み(`docs/ASSET_GUIDELINES.md` の手順)は Phase と独立して随時実施する。素材が届き次第、暫定素材(コード描画)から差し替える。

## 5. 品質ルール

- `core/` の変更には必ず単体テストを付ける(`npm test` で実行)
- UI 変更時はブラウザでの動作確認結果(スクリーンショット/動画)を PR に添付する
- 乱数はシード注入可能にし、抽選確率はシミュレーションテスト(大量試行)で検証する
