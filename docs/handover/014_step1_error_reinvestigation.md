## 事実(両ランに共通)

- 環境セットアップは成功(INSTALL_SUCCEEDED)。実行された全ツールコール(ファイル読み取り・ターミナル)は正常終了(exit 0)。
- **コード変更・コミット・プッシュ・ブランチ作成はゼロ**。両ランとも `HANDOVER.md` → `ROADMAP.md` → `SPEC.md` → `reel.ts` 等を読む調査フェーズのみで終了。**失われた成果物はない**。
- 1 回の応答で **5〜6 万文字級の思考(thinking)ブロック**を生成し、その生成に **12〜26 分**を要するサイクルを繰り返した。1 回目のランでは 5 万字超の思考が 4 回(計約 23 万字)続き、毎回ほぼ同じ実装計画を再導出して実装に着手できなかった。
- 終了の仕方も同一: 最後のツール実行が正常に完了した後、次の応答生成が返らないままトランスクリプトが途切れ、
  - 1 回目: 最終ツール成功 13:09:09 → 最終活動 13:39:09(**30 分間** 生成リトライが失敗し続けた形跡)
  - 2 回目: 最終ツール成功 20:50:39 → 最終活動 21:05:39(**15 分間** 同上)
  で ERROR 確定。UI 表示の「unexpected error repeatedly(繰り返し失敗)」と整合する。
- 2 回目のランでは、引数が空の壊れたツールコール(`grep` 引数なし、`todo_write` が `success: false`)も記録されており、生成が劣化していた兆候がある。

## 結論(原因)

リポジトリのコード・環境・プロンプトの誤りではない。**STEP 1 の巨大なスコープ(SPEC 全読解 + リール配列/ライン/停止制御の全面書き換え + テスト再設計)を 1 応答で計画しようとした結果、モデル(claude-fable-5-thinking-high)の思考生成が病的に肥大化し、応答生成が繰り返し失敗して Cursor Cloud 側でランが打ち切られた**もの。ラン 013 時点の結論「基盤の異常終了・リポジトリ側の修正不要」を裏付けつつ、トリガーは「タスクの粒度が大きすぎること」と特定した。

## 再発防止策(次に STEP 1 を依頼するときの推奨)

1. **STEP 1 を小さなサブタスクに分割して 1 つずつ依頼する**(1 ラン 1 サブタスク)。例:
   - 1a: リール配列データの差し替え + 図柄 8 種化(`reel.ts` のデータ定義と `roles.ts` 整合、既存テストの最小修正まで)
   - 1b: 有効ライン横 3 + 斜め 2 のライン判定と入賞判定
   - 1c: 停止制御(リプレイ 100% 引き込み / 押し順ベル / DDT 黒バー / リーチ目 7 揃い取りこぼし)
   - 1d: `src/assets/index.ts` の `SYMBOL_IMAGES` 仮割り当て解消 + UI 確認
2. 依頼文に「**まずブランチを作成し、事前調査は HANDOVER・ROADMAP と該当 SPEC セクションの確認に留め、早い段階で小さくコミット・プッシュすること**」を明記する。
3. 同じ現象(長時間無応答→ERROR)が再発する場合は、**モデルを変えて再試行**するのも有効。
4. 再試行は安全(両ランともリポジトリへの副作用なし)。
