## 前回(AGENT #031 初版)からの仕様変更点

1. **V ストックはバトル開始時に先に消化**(確定 29。旧 = 継続率抽せんに漏れたときのみ消費)→ ストックの温存がなくなり平均連チャンが低下。
2. **10 連目のセットも敗北あり**(確定 30。旧 = 必ず移行)→ 上位 AT 移行・エンディング完走率が低下。
3. **10 連目の勝利でエンディング 10G(押し順ナビあり)を経由**(確定 30・31。旧 = 直接移行 + エンディング暫定 1G)→ エンディング分の純増が増加(1 回あたり 10G × 約 +6.4 枚)。

## 計測方法

- 実行コマンド: `npm run sim -- <ゲーム数> <シード>`(`scripts/run_simulation.ts` → `src/core/simulate.ts`)
- 1G の実行: `src/core/play.ts` の `playGame` = 役抽せん(`drawRole`)→ 打ち方ポリシー → リール制御(`resolveSpin`)→ ステートマシン(`advanceGame`)
- 打ち方(確定 26):
  - 通常時 = **左第一・適当押し**(押下位置は全リール一様ランダム。チェリー・スイカ・リーチ目は取りこぼしあり)
  - AT 中・エンディング中 = **ナビ遵守**(押し順ベルは中第一 = 斜め揃い 13 枚。ベル以外は左第一・適当押しのままなのでレア役の取りこぼしは通常時と同様に発生)
- 区間の分類(ゲーム開始時点のフェーズ):
  - 通常区間 = 通常 / 前兆 / 連続演出(連続演出成功の告知ゲームまで)
  - AT 区間 = AT(小役・バトル)/ エンディング
- 指標の定義は `src/core/simulate.ts` のヘッダーコメント参照(機械割 = 総払出 ÷ 総投入、コイン持ち = 通常区間 50 枚あたりゲーム数、初当り = 通常区間 G 数 ÷ AT_START 回数、連チャン数 = セット数)

## ユーザーへの確認依頼(確定 26 のフォローアップ)

初版レポートへの回答「とりあえずはこのバランスで問題ない(ただし確定 29〜31 に合わせる)」を受けて反映した結果、出玉バランスが変化しました:

3. 通常時純増 −1.86 枚/G・コイン持ち 26.9G は初版から変化なし(想定どおり)。
