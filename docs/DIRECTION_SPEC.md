## 0. ユーザー指示による確定内容(2026-07-13 = SPEC 確定 33)

1. **通常 4 背景(義経 / 静 / 弁慶 / 夕方)**: 背景固有の予告演出 5 種 + 背景共通の予告演出 4 種は、
   それぞれ**弱パターンと強パターンの 2 バリエーション**を持つ。
2. **前兆背景**: 固有予告 1・2・3 は**前兆期待度の差**(1 = 弱 / 2 = 中 / **3 = 本前兆確定**)。
3. **AT 中・上位 AT 中(小役パート)**: 固有 1 = **ベルナビ演出**(押し順を示唆)/
   固有 2 = **レア役示唆** / 固有 3 = **強予告(V ストック濃厚)**。
4. **予告は全てムービー素材で持つ**。小役示唆系の演出は最終的に画面へベル等の図柄画像を表示する。
5. 演出は**仮素材で進める**(実素材の差し替えは STEP 4f)。

## 2. シナリオテーブルのスキーマ設計

確定 28(シナリオ方式)に基づき、**前兆当せん時(前兆種別・総 G 数・連続演出種別の確定時)に
シナリオを一括抽せん**し、毎 G はシナリオ参照で表示する。実装は STEP 4b(`src/core/scenario.ts` 新設)。

### 2.1 設計の要点

- シナリオは**背景に依存しない抽象表現**で持つ(スロット + 強度レベル)。前兆中の滞在背景は
  当せんの次ゲーム(背景移行契機 2/4)まで確定しないため、**具体的なムービーの解決は表示時**
  (`direction.ts`)に「現在の背景 × スロット × レベル」で行う。
- **強度レベルは 4 段階**: `L0`(予告なし)/ `L1`(弱)/ `L2`(強)/ `L3`(確定。**本前兆のみ**出現)。
  - 通常 4 背景での解決: L1 → スロットの弱パターン / L2・L3 → スロットの強パターン
    (通常背景に確定パターンはないため L3 は強で表示)。
  - 前兆背景での解決: L1 → `ZENCHO_1` / L2 → `ZENCHO_2` / L3 → `ZENCHO_3`(スロットは無視)。
- `GameState` の骨格(2a)は変えない。`OmenPhase` / `RenzokuPhase` への**フィールド追加のみ**で
  シナリオ情報を持たせ、毎 G 出力(`AdvanceResult`)へ「この G の演出(スロット・レベル)」を含める。

### 2.2 型定義案(4b で実装)

```ts
/** 予告スロット(背景非依存の抽象 ID) */
export type YokokuSlot =
  | { kind: 'KOYU'; index: 1 | 2 | 3 | 4 | 5 }    // 背景固有予告 1〜5
  | { kind: 'KYOTSU'; index: 1 | 2 | 3 | 4 };     // 背景共通予告 1〜4

/** 強度レベル(L0 = 予告なし / L1 = 弱 / L2 = 強 / L3 = 確定。L3 は本前兆のみ) */
export type ScenarioLevel = 0 | 1 | 2 | 3;

/** 前兆 1G 分の演出 */
export interface ScenarioStep {
  level: ScenarioLevel;
  /** level >= 1 のとき使用(前兆背景では無視される) */
  slot?: YokokuSlot;
}

/** 連続演出 1G 分の演出(1〜3G 目。4G 目は成否告知) */
export type RenzokuStepPattern = 'NORMAL' | 'CHANCE';

/** 前兆当せん時に一括抽せんされるシナリオ */
export interface OmenScenario {
  /** 前兆 1G 目〜総 G 数(7〜10)の演出列 */
  steps: readonly ScenarioStep[];
  /** 連続演出 1〜3G 目のチャンスアップ(4G 目は成否告知で固定) */
  renzokuSteps: readonly [RenzokuStepPattern, RenzokuStepPattern, RenzokuStepPattern];
}
```

- 抽せん関数: `drawOmenScenario(rng, kind, totalGames): OmenScenario`。
  各 G 独立に「レベル → スロット」の順で抽せんする(乱数消費順を固定しテストで検証)。
- 配線位置: `state.ts` スロット 3 の前兆スケジュール抽せん(`scheduleOmen`)直後
  (`initGameState`・AT 終了時の本前兆リドローを含む)。
- **偽→本書き換え時はシナリオも引き継ぐ**(再抽せんなし = Q16 推奨。確定 21 の a/b と整合)。

### 2.3 AT 中(小役パート)予告の決定方式

小役パート予告はシナリオ方式ではなく**毎 G 抽せん(成立役ベース)**とする(Q17 推奨)。

- 決定タイミングはレバーオン時(役抽せん直後)。ベルナビ・レア役示唆は停止前に見せる意味がある
  (ナビ遵守・目押し補助)ため。
- `AT_STRONG`(V ストック濃厚)は「ストック獲得率の高い成立役」でのみ出現させることで
  「濃厚」の看板を守る(実際のストック抽せん結果とは独立の示唆。抽せん自体は従来どおり
  `advanceGame` 内で行い、演出と矛盾しない)。
- バトルパート中はバトル演出(2.5)が優先。小役パート予告は出さない。

### 2.4 連続演出 A/B/C の 4G 構成

- G1 = 導入 / G2 = 展開 / G3 = あおり / G4 = **成否告知**(勝利 → AT 突入・敗北 → 通常へ)。
- G1〜G3 は各 G「通常 / チャンス」の 2 パターン(シナリオの `renzokuSteps` で決定)。
- A / B は背景固有(発展時の滞在背景の素材を使う)、C は背景共通。
- 偽→本書き換えが演出中に起きた場合も演出は継続し、4G 目の告知が勝利になる(確定 21(c))。

### 2.5 バトルパート演出(8G)の決定方式

バトル演出は**バトル開始時(バトル 1G 目)の継続確定状態でルートを一括抽せん**し、
8G 目の復活判定のみ実際の結果で表示する(Q18 推奨)。

- **開始時に継続確定済み**(V ストック消費 = 確定 29 / 継続率当せん): 勝利ルート
  (義経攻撃系 or 頼朝攻撃 → 耐える)を振分け抽せん。8G 目は「義経台詞で次セットへ」。
- **開始時に未確定**: 敗北寄りルート(頼朝攻撃 → 耐えれない)で進行し、8G 目の
  第 3 リール停止時に「その時点の継続確定状態」で復活成否を表示
  (バトル中の小役継続抽せんで確定していれば復活 = 継続 / 未確定のままなら敗北 = AT 終了)。
- チャンスアップ(1〜3G 目の通常/チャンス)は各 G 独立に抽せん(継続確定側で出やすい)。
- 上位 AT はルートに「ダブル攻撃」(確定済みのみ)を加えた共闘版で同じ仕組み。

## 5. 質問リスト(Q12〜Q20。推奨案付き)

Q1〜Q11(STEP 2 着手前)からの連番。**回答があるまでは推奨案(仮テーブル)で実装を進め、
回答があり次第データ差し替えで追随する**(STEP 2 と同じ方式)。

- **Q12. 通常時(非前兆)の予告発生**: 前兆中でないゲームにも単発でガセ予告を出すか?
  **推奨: 出さない**(ガセの役割は偽前兆(7〜9G + 連続演出失敗)が担っており、
  単発ガセを足すと「予告が出た = 前兆中」の法則が崩れて偽前兆の存在感が薄れる。
  欲しくなったら後から低頻度で追加できる構造にはしておく)。
- **Q13. 強度レベル体系**: 予告の強度を「なし / 弱 / 強 / 確定」の 4 段階とし、
  通常 4 背景 = 弱・強の 2 素材(確定は強で表示)/ 前兆背景 = 固有 1(弱)・2(中 = レベル強に対応)・
  3(確定)へマッピングする設計でよいか? **推奨: これで進める**
  (前兆背景の「中」= 通常背景の「強」相当。確定(L3)は本前兆でのみ抽せんされるため、
  前兆背景の固有 3 = 本前兆確定(確定 33)が自然に成立する)。
- **Q14. 予告スロット振分け**: 固有 1〜5 = 各 12% / 共通 1〜4 = 各 10% の仮値でよいか?
  **推奨: 仮値で開始**(見た目の頻度の問題で出玉に影響しないため、実機で見て調整でよい)。
- **Q15. 強度カーブ**: 3.1 の仮テーブル(偽は確定なし・終盤ほど強い・本は強が多い)でよいか?
  **推奨: 仮値で開始**(本/偽の見分けがつきすぎないよう、偽にも強を最大 25% 残している)。
- **Q16. 偽→本書き換え時のシナリオ**: 書き換え時(確定 9・21)にシナリオ(演出の並び)も
  そのまま引き継ぐか、残り G を本前兆カーブで再抽せんするか?
  **推奨: そのまま引き継ぐ**(確定 21 の「G 数・連続演出とも引継ぎ・再抽せんなし」と整合し、
  演出の急変で書き換えがバレない。書き換え後もどうせ連続演出は成功する)。
- **Q17. AT 中予告**: 決定方式(毎 G・成立役ベース = 2.3)と振分け(3.4)でよいか?
  また `AT_RARE`(レア役示唆)の最終表示は**具体的な図柄画像**(スイカ / チェリー等)でよいか、
  それとも「レア役?」の汎用表示か? **推奨: 3.4 の仮値 + 具体図柄を表示**
  (AT 中のレア役はストック源なので、目押し補助として具体役を見せる価値が高い)。
