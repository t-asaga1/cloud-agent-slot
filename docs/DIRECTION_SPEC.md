## 2. シナリオテーブルのスキーマ設計

予告は **2 系統**(確定 34): (A) **小役示唆予告** = 毎 G・成立役ベースの独立抽せん /
(B) **前兆シナリオ予告** = 確定 28(シナリオ方式)に基づき**前兆当せん時
(前兆種別・総 G 数・連続演出種別の確定時)に一括抽せん**し、毎 G はシナリオ参照で表示。
実装は STEP 4b(`src/core/scenario.ts` 新設)。

### 2.1 設計の要点

- **(A) 小役示唆予告(固有 1〜3 / 共通 1・2)**: 「成立小役が決定した後にどの演出を出すか」
  (確定 34)。レバーオン時(役抽せん直後)に成立役から独立関数 `drawKoyakuHint` で抽せんする
  (`advanceGame` の乱数列は消費しない)。通常時(非前兆)にも出す(Q12)。
  強度は成立役に応じた弱 / 強(強いほどレア役寄り)。
- **(B) 前兆シナリオ予告(固有 4・5 / 共通 3・4)**: シナリオは**背景に依存しない抽象表現**で持つ
  (スロット + 強度レベル)。前兆中の滞在背景は当せんの次ゲーム(背景移行契機 2/4)まで
  確定しないため、**具体的なムービーの解決は表示時**(`direction.ts`)に
  「現在の背景 × スロット × レベル」で行う。**通常時(非前兆)には出さない**(Q12。
  「前兆系の予告が出た = 前兆中」の法則)。
- **強度レベルは 4 段階**(Q13): `L0`(予告なし)/ `L1`(弱)/ `L2`(強)/ `L3`(確定。**本前兆のみ**出現)。
  - 通常 4 背景での解決: L1 → スロットの弱パターン / L2・L3 → スロットの強パターン
    (通常背景に確定パターンはないため L3 は強で表示)。
  - 前兆背景での解決: L1 → `ZENCHO_1` / L2 → `ZENCHO_2` / L3 → `ZENCHO_3`(スロットは無視)。
- 同一 G の競合(表示時の規約 = 4c): 前兆中にシナリオ予告(L1 以上)がある G は
  シナリオ予告を優先し小役示唆予告は出さない。シナリオ L0 の G と通常時は小役示唆予告を出せる。
  前兆背景滞在中は小役示唆予告なし(前兆背景の固有 1〜3 は期待度ラダー専用のため)。
  **こぼすベル(左第一こぼし抽せんで 12/13 を引いたゲーム = SPEC 確定 35)にも小役示唆予告は
  出さない**(揃うベル = 1/13 には出す。2026-07-14 ユーザー指示。UI がレバーオン時の
  bellMiss フラグで `drawKoyakuHint` の抽せん自体をスキップする)。
- `GameState` の骨格(2a)は変えない。`OmenPhase` / `RenzokuPhase` への**フィールド追加のみ**で
  シナリオ情報を持たせる(毎 G 出力 `AdvanceResult.state.phase` から参照できる = 確定 28)。

### 2.2 型定義(4b で実装済み。`src/core/scenario.ts` が正)

/** 前兆 1G 分の演出 */

  各 G 独立に「レベル → (L1 以上のとき)スロット」の順で抽せんし、最後に連続演出
  チャンスアップ 3 つを抽せんする(乱数消費順を固定しテストで検証)。

  (`initGameState`・AT 終了時の本前兆リドローを含む)。`OmenPhase.scenario` に保持し、

- **偽→本書き換え時はシナリオも引き継ぐ**(再抽せんなし = Q16 回答。確定 21 の a/b と整合)。

  (UI がレバーオン時などに別 rng で呼ぶ = 「6.」)。

### 2.3 AT 中(小役パート)予告の決定方式

小役パート予告はシナリオ方式ではなく**毎 G 抽せん(成立役ベース)**とする(Q17 回答 = 推奨どおり)。

- 決定タイミングはレバーオン時(役抽せん直後)。ベルナビ・レア役示唆は停止前に見せる意味がある
  (ナビ遵守・目押し補助)ため。
- `AT_RARE`(レア役示唆)の最終表示は**具体的な図柄画像**(スイカ / チェリー等 = Q17 回答)。
- `AT_STRONG`(V ストック濃厚)は「ストック獲得率の高い成立役」でのみ出現させることで
  「濃厚」の看板を守る(実際のストック抽せん結果とは独立の示唆。抽せん自体は従来どおり
  `advanceGame` 内で行い、演出と矛盾しない)。
- バトルパート中はバトル演出(2.5)が優先。小役パート予告は出さない。

### 2.4 連続演出 A/B/C の 4G 構成

- G1 = 導入 / G2 = 展開 / G3 = あおり / G4 = **成否告知**(勝利 → AT 突入・敗北 → 通常へ)。
- G1〜G3 は各 G「通常 / チャンス」の 2 パターン(シナリオの `renzokuSteps` で決定)。
- A / B は背景固有(発展時の滞在背景の素材を使う)、C は背景共通。
- 偽→本書き換えが演出中に起きた場合も演出は継続し、4G 目の告知が勝利になる(確定 21(c))。

### 2.5 バトルパート演出(8G)の決定方式

バトル演出は**バトル開始時(バトル 1G 目)の継続確定状態でルートを一括抽せん**し、
8G 目の復活判定のみ実際の結果で表示する(Q18 回答で確定)。
**各分岐にそって約 20 ルートを作成し、勝利時・敗北時・復活時および継続率でそれぞれ振分けを持つ**
(Q18 回答。ルート表と振分けは 3.6)。

- **開始時に継続確定済み**(V ストック消費 = 確定 29 / 継続率当せん): **勝利ルート**
  (義経攻撃系 or 頼朝攻撃 → 耐える)を**継続率別の振分け**で抽せん。8G 目は「義経台詞で次セットへ」。
- **開始時に未確定**: **敗北寄りルート**(頼朝攻撃 → 耐えれない)を**継続率別の振分け**で抽せんして
  進行し、8G 目の第 3 リール停止時に「その時点の継続確定状態」で復活成否を表示
  (バトル中の小役継続抽せんで確定していれば**復活告知を復活時振分けで抽せん** = 継続 /
  未確定のままなら敗北 = AT 終了)。
- チャンスアップ(1〜3G 目の通常/チャンス)は**ルートへ焼き込み**(ルート自体がチャンスアップの
  バリエーションを持つ = Excel のパターン No 採番と整合)。
- 上位 AT はルートに「ダブル攻撃」(確定済みのみ)を加えた共闘版で同じ仕組み(17 ルート)。
- **実装解釈(STEP 4e)**: バトル開始時の継続率抽せん(確定 29)は core ではバトル 1G 目の
  `advanceGame`(= 全停止時)で行われるため、レバーオン時点で UI が知り得る「開始時確定」は
  V ストック(> 0 なら消費で確定)のみ。ルートはバトル 1G 目のレバーオンで
  「確定 = V ストック有無」により仮抽せんし、**1G 目の全停止で率当せん(継続確定)が判明したら
  勝利ルートへ抽せんし直す**(2G 目以降の表示から反映)。2G 目以降の小役継続当せんでは
  引き直さない(敗北寄りルートのまま 8G 目の復活告知で見せる)。
