## 2. 演出の共通文法(4 背景で統一する提案)

1. **構成 = 「静止 → 気づき/所作(1 アクション)→ カメラ移動 → 受け(図柄表示スペース)」**。
   ムービーの終盤は「キャラが画面から外れた(または引いた)空きスペース」で終わり、
   そこへアプリ側が成立役の図柄画像を表示する(図柄はムービーへ焼き込まない =
   DIRECTION_SPEC「4.」の方針を維持 = Q23 承認。仮素材と同じ表示機構がそのまま使える)。
   **図柄はフェードインではなく、PAN が終わった時点で「すでに映っている」形で即時表示**
   (2026-07-16 承認時の追加指示。実装は `KOYAKU_HINT_PRESENTATION` の `symbolDelayMs`)。
2. **弱 / 強の作り分け = 「弱 = 左へ / 強 = 右へ」で全背景統一**(ユーザー例の一般化)。
   - PAN・視線・所作の方向を左右で反転させるだけの「微妙な違い」に揃える。
   - 遊技者が覚えられる隠し法則になり、かつ生成プロンプトの差分が 1 語で済む
     (制作・検証コストが最小)。
   - 方向以外の追い差分(光の量・エフェクトの強さ)は強側にだけ薄く足す(下記各案)。
3. **尺 = 6 秒目安**・720p(1280×720)・音声なし(SE は別管理)。
   1 変動の最短秒数がウェイトありで 4.1 秒のため 8 秒は若干長い、との
   2026-07-16 ユーザー判断で 6 秒を目安とする。図柄表示は終盤の PAN 完了時点
   (現行実装の一律 1.5 秒 → スロット別秒数へ変更済み。固有 1 = 4.6 秒)。
4. **背景はムービーへ焼き込む**(アルファ付き動画は Seedance では作れない)。
   既存の背景ループ動画の見た目と合うよう、`incoming/reference/背景/` の参考画像
   (+ 必要ならループ動画の代表フレーム)を背景参照として渡す。
   ループ → 予告の切り替わりはカットイン(瞬時切替)で開始し、開始フレームを
   背景と同構図にして違和感を抑える。

## 3. 各背景の固有予告 1 の案

Seedance 適性の凡例: ◎ = 得意分野そのもの / ○ = 問題なし / △ = リスクあり(対策込みで可)。

### 3.1 義経背景(鞍馬寺)—「気配」(ユーザー例の採用)

- **本命案**: 鞍馬寺の山門前、義経が正面を向いて静かに立つ。風で木の葉が横切る →
  何かに気づいて**ゆっくり横を向く** → カメラが視線の方向へ PAN → 山道の空きスペースで
  ムービー終了 → 図柄フェードイン。
  - 弱 = 左を向く + 左 PAN(葉が数枚舞う)/ 強 = 右を向く + 右 PAN(**桜吹雪が強めに舞う**)。
  - Seedance 適性: **◎**。1 キャラ・小さな 1 アクション・ゆっくりした PAN は最も成功率が高い形。
    振り向きは「ゆっくり」指定で顔ドリフトを回避(SEEDANCE_GUIDELINES 5.)。
- 代替案「抜刀の構え」: 柄に手を掛け、弱 = 鯉口を切るだけ / 強 = 半分抜いて刃が光る →
  カメラが刀身へ寄る → 刃の輝きに図柄。参考画像あり(義経抜刀 1/2)。適性 ○
  (動きは小さいが、弱強差が方向でなく所作の深さになり統一文法から外れる)。

### 3.2 静背景(清水寺)—「舞の一差し」

- **本命案**: 清水の舞台、静が扇を胸元に構えて立つ → **扇をすっと差し伸べる 1 所作**
  (舞の一動作)→ 花びらが扇の指す方向へ流れ、カメラが同方向へ PAN → 空(舞台の外の景色)で
  ムービー終了 → 図柄フェードイン。
  - 弱 = 左へ差す + 左 PAN(花びら少量)/ 強 = 右へ差す + 右 PAN(**花びら + 金色の光の粒**)。
  - Seedance 適性: **◎**。優雅でゆっくりした腕の所作 + 花びら・光のパーティクルは
    動画生成モデルの得意分野。顔は正面キープで顔ドリフトも起きにくい。
- 代替案「祈り」: 両手を合わせて祈ると光の玉が昇る → カメラが上へティルト → 光の中に図柄。
  弱 = 白い光 / 強 = 金色の光。適性 ◎(最小動作)だが、弱強差が「色」になり
  左右統一文法から外れる。

### 3.3 弁慶背景(比叡山の滝)—「石突の一突き」

- **本命案**: 滝を背に弁慶が薙刀を携えて立つ → **薙刀の石突を地面にドンと突き立てる**
  (上下の 1 アクション)→ 水しぶきが舞い、カメラが横へ PAN → 滝つぼ横の岩場の
  空きスペースでムービー終了 → 図柄フェードイン。
  - 弱 = 左 PAN(しぶき小)/ 強 = 右 PAN(**しぶきが大きく割れて虹がかかる**)。
  - Seedance 適性: **◎**。上下の単純動作はモーション破綻しにくく、頭部がほぼ固定で
    顔も安定。水しぶき・水流の表現は得意分野。
- 代替案「薙刀一閃」: 横薙ぎで水煙を割る。迫力はあるが高速アクション = 顔ドリフト・
  破綻リスクがあるため、採用するならワイドショット(顔を大きく写さない)で作る。適性 △〜○。
  ※一閃は強アクションとして固有予告 2 以降(より期待度の高い枠)へ温存するのも手。

### 3.4 夕方背景 —「すれ違い」(義経 + 弁慶)

- **本命案**: 夕陽を背にした**逆光シルエット気味**の画。弁慶が立ち、義経がその横を
  ゆっくり通り過ぎる(すれ違い)→ 片方が立ち止まる → カメラが夕陽へ PAN →
  夕陽の空きスペースでムービー終了 → 図柄フェードイン(夕陽に重なる)。
  - 弱 = 義経だけが立ち止まって振り返る + 左 PAN / 強 = **二人同時に立ち止まって
    振り返る** + 右 PAN。
  - Seedance 適性: **○**。2 キャラは参照枚数が上限(顔 + 全身 × 2 = 4 枚 + 背景 1 = 5 枚)に
    達するが、**夕方の逆光シルエット表現なら顔の同一性への要求自体が下がる**ため、
    4 背景の中で唯一の 2 人予告としては最も安全な作り。二人の同時動作の整合が
    乱れた場合はカット割り(手前弁慶ナメ + 奥義経)へ逃がす。
- 代替案「背中合わせ」: 二人が背中合わせに立ち、同時に顔を上げる → カメラが引いて空へ。
  動きが最小で適性 ◎だが、弱強差がカメラの寄り/引きになり左右統一文法から外れる。

## 4. Seedance での生成手法(弱強の「微妙な違い」の担保)

「強は弱の微妙なバリエーション違い」を生成物で成立させるには、**2 本の出だしを一致させて
途中から分岐させる**のが確実。以下の手順を推奨:

1. **起点静止画を背景ごとに 1 枚作る**(GPT Image 2 `edit`・quality=high):
   設定資料(顔 + 全身)+ 背景参考画像から「キャラが定位置に立つ予告開始構図」の静止画を生成。
   顔・衣装は設定資料から変えない指示を固定で含める(`gen_character_sheets.mjs` と同方式)。
2. **Seedance `reference-to-video` へ「起点静止画 = 最初のフレーム」と明示して弱・強を生成**:
   - 参照の並び: `@Image1` 顔 → `@Image2` 全身 → `@Image3` 起点静止画(first frame 指定)
     (夕方のみ 顔 ×2 + 全身 ×2 + 起点 = 5 枚。SEEDANCE_GUIDELINES の上限目安内)。
   - プロンプトは弱・強で**方向の 1 語 + 強側のエフェクト 1 句のみ**を変える。
   - これで 2 本の開始が完全一致し、「途中から分岐する微妙な違い」になる。
     API に seed 指定があれば併用して揺れをさらに抑える。
3. **下書き → 本番の 2 段階**: 構図・動きの確認は `fast`/480p、採用版は standard/720p
   (SEEDANCE_GUIDELINES 4. のとおり。`generate_audio` = false)。
4. **生成後の検証**: 設定資料と並べた顔の目視検証 + videoReview(顔ドリフト・破綻・
   弱強の差分が意図どおりかを確認)。NG は参照の渡し方・プロンプトを直して再生成。
5. **取り込み**: 採用動画を `incoming/` 直下へ → `scripts/import_incoming_assets.py` で
   WebM(VP9)720p 変換 → 既存仮素材と同名で差し替え → `manifest.json` を user-provided +
   生成ツール名で登録(ASSET_GUIDELINES)。

コスト目安(fal.ai 2026-07 時点): 本番 standard 720p $0.3034/秒 × 5 秒 × 8 本 ≒ **$12** +
下書き(fast 480p)分。生成時間は 1 本あたり数分〜10 分程度(#072・#073 実測ベース)。

### Seedance の得意・不得意(案の採否基準)

- **得意(今回の案は全てここに寄せた)**: 1 キャラ・1 アクション / ゆっくりした PAN・ドリー・
  ティルト / 風・花びら・水しぶき・光などの自然エフェクト / 逆光・シルエット。
- **不得意(避けた)**: 高速アクション・激しい頭部回転(顔ドリフト)/ 複数キャラの接触・
  絡み / 文字・図柄の描画(→ 図柄はアプリ側オーバーレイで回避済み)/ 長尺の一貫性。

## 8. 固有予告 1 — 残り 3 背景(静・弁慶・夕方)のプロンプト案

生成手法は義経版と完全に同じ(「4.」の手順 = 起点静止画 → 弱・強を分岐生成。
`scripts/gen_yokoku_koyu1.mjs` の `BACKGROUNDS` へ以下の設定を追記して実行する)。
演出内容は承認済みの本命案(「3.2」〜「3.4」= Q24)どおり。
義経版の教訓を反映済み: **強側のエフェクトは「冒頭から最後まで大量に舞い続ける」と
明示的に強調しないと描画されない**(#075 の実測)。

### 12.1 指示書のフォーマット(AGENT #079 の回答 = パワーポイント推奨)

ユーザーからの質問「指示書はパワーポイントとエクセルどちらで作るのが良いか」への回答:

- **パワーポイント(pptx)推奨**。理由:
  - **1 スライド = 1 画像(または 1 予告)**にすると、説明文と参考画像・ラフスケッチの
    対応がスライド単位で一意になり、読み取りの取り違えが起きない。
  - 構図・レイアウト(キャラの位置、図柄表示スペースの位置取りなど)は
    スライドに絵や図形で直接描けるため、画像制作の指示に向いている。
  - VM 上で python-pptx により全スライドのテキスト + 貼付画像(元解像度のまま)を
    抽出できることを実機検証済み(AGENT #079)。
- エクセルでも読み取りは可能(openpyxl。セル値 + 貼付画像とそのアンカー位置を抽出できる
  ことも実機検証済み)なので、**表形式(一覧管理)が作りやすければエクセルでも問題ない**。
  数値テーブル(振分け値など)を渡す場合は従来どおりエクセルが確実。
- **1 スライドに入れてほしい項目(推奨テンプレート)**:
  1. 予告名(例: 義経背景 固有予告 1 弱)と画像番号(例: 1/3)
  2. シーン説明(何が起きている絵か)
  3. 登場キャラ(登場キャラルール「0.」準拠: 義経/静/弁慶背景 = 背景キャラのみ・
     夕方 = 義経 + 弁慶のみ)
  4. 構図・カメラ(引き/寄り、キャラの位置)
  5. 図柄表示スペースの位置(アプリ側で成立役の図柄を重ねる場所)
  6. 弱/強(パターン間)の差分
  7. 参考画像・ラフスケッチの貼付(あれば)
  8. NG 事項・備考
- **入稿方法**: `incoming/specs/` へ pptx(または xlsx)をコミット & プッシュ
  (従来の仕様書入稿と同じ。`incoming/README.md` 参照)。画像を多く貼ると 25 MiB
  (ブラウザアップロード上限)を超えることがあるので、その場合は GitHub Desktop で
  プッシュする。チャット添付でも読めるが、次の AGENT 以降も参照できるよう
  リポジトリ入稿を推奨。
