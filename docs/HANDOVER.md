## 2. ルール(2026-07-18 ユーザー指示で変更)

- **「1 AGENT 1 やり取り」ルールは廃止**。同一 AGENT と複数回やり取りしてよい(プロンプト承認 → 生成 → 確認 → 組込みを 1 会話で進めてよい)。
- **画像作成・実装・検証はできるだけ簡易に進める**。
- **検証はアプリ本体での確認を主とする**: `npm test` + 必要最小限のブラウザ確認で足りる。**録画・スクリーンショットなどユーザー確認用資料の作成は最小限にする**(凝ったデモ動画は作らない)。
- **画像の生成にはリファレンス(参考画像)を必ず使う**(下記 4. 参照)。
- 素材生成の 3 原則は継続: (1) 生成前にプロンプト・構図案をユーザーへ提示して承認を得る / (2) 生成後は組込み前にユーザーへ確認してもらう(**原本は同ランで `incoming/` へコミットして現物確保**)/ (3) 指示なしの再生成禁止(課金)。
- HANDOVER の更新は「引継ぎに影響する変更があったとき」でよい。履歴コピー(`docs/handover/NNN_*.md`)は大きな区切りのみ。

## 4. 画像生成のルール・ノウハウ

- **リファレンス必須**: キャラは `incoming/reference/設定資料/<名前>_{顔,全身}.png` のペアを、背景は `incoming/reference/背景/` の参考画像を参照に渡す。**連続する画像は前の生成画像を参照に連鎖**させて構図・キャラの一貫性を保つ。キャラデザインの禁止事項(義経 = ポニーテール禁止など)はプロンプトへ明文化する(実例 = `scripts/gen_battle_images.mjs` / `gen_yokoku_kaiwa.mjs`)。
- **生成手段**: fal.ai の `openai/gpt-image-2/edit`(Secret `FAL_KEY2`・quality=high・16:9 は 1792x1008)か、内蔵 GenerateImage(課金なし。ただし設定資料からのドリフトが起きやすい)。
- **文字は画像に焼き込まない**(台詞・技名・数値はアプリ側テキスト描画)。図柄合成する画像は「中央を空ける」旨をプロンプトへ明記。
- **原本は同ランで `incoming/` 配下へコミット**(アーティファクトはラン終了で消える)。**チャット添付画像は VM で取得できない**ため、参照が必要な画像は `incoming/` への main プッシュで入稿してもらう。
- content_policy_violation が出たら (1) 負傷・苦痛の直接描写を削る → (2) そのままリトライ(モデレーションは確率的)。
- 登場キャラのルール: 義経/静/弁慶背景 = 背景キャラのみ・夕方背景 = 義経と弁慶のみ。実在機種の素材・画像の流用禁止(pptx の参考キャプチャを fal.ai へ渡さない)。

## 5. 組込みの定型パターン(紙芝居)

静止画演出の組込みは次の最小手順で済む(実例 = 経緯履歴 `docs/handover/093_at_battle_integration.md` 等):

1. 原本 PNG を 16:9 クロップ + 1280x720 WebP 化して `src/assets/images/<用途>/` へ。
2. `manifest.json` へ登録(出所 = 生成スクリプト名)。
3. `src/ui/direction.ts` の対応表(`KOYAKU_HINT_STILLS` / `atBattleStill` など)へエントリ追加。表示切替は既存の `stoppedReels` 検知(レバオン → 第 1 停止 → 第 3 停止)が使える。
4. 存在検証テスト(direction.test.ts)へキー追加 → `npm test`。
5. `incoming/` の原本を削除(Git 履歴に残る)。

## 6. 実装上の注意(以後も維持する規約)

- **乱数消費順序を変えたらテストを取り直す**: `playGame` / レバーオンの順序は「役(赤7待機中は消費なし)→ ベルこぼし → ナビ押し順 → 押下位置 3 → `advanceGame` 内部」。変えると `flow.test.ts` のシード・`simulate.test.ts` の固定値が崩れる(意図した変更ならシードと固定値を取り直す)。
- **演出用の乱数は UI 専用 rng(`hintRng`)**を使い、出玉に影響する `advanceGame` の乱数列と分離する。
- **SE は `playCue(SoundCueId)`(`src/ui/sound.ts`)経由・BGM は `bgmTrackForState`(`src/ui/bgm.ts`)経由**。ファイルを UI から直接参照しない(差し替えを対応表 1 箇所で完結させる)。
- **演出の中身(文言・画像・時間・SE)は `src/ui/direction.ts` に集約**し、`DirectionLayer` は描画専用に保つ。
- **メーター・遊技データの更新はレバーオンと全停止の 1G 締めの 2 タイミングのみ**(`meterOnLever` / `meterOnFinish` / `statsOnFinish`)。
- **数値(確率・配当・配列)は SPEC.md と `src/core/` を必ず一致させる**。抽せんテーブルは分母付き整数で、行合計 = 分母をテストが検証している。
- ブラウザ API(音声・動画・永続化)は `src/platform/` のラッパー経由(exe 対応のため)。
- 素材追加時は `manifest.json` に出所を登録(`docs/ASSET_GUIDELINES.md`)。
- PR は base を `main` にして 1 本ずつマージ(本ファイルが競合しやすい)。同じタスクを複数エージェントへ並行依頼しない。

## 7. デバッグ・検証のコツ

- 状態の再現は**デバッグパネル(サイドパネル内)の「成立役の強制指定」+「1G消化(オート)」ボタン**で行う(例: 中段チェリー強制 → 本前兆 → AT。ハズレ強制 = 抽せんなしで G 消化)。連続オート消化(160ms/G)は狙ったフェーズで止めにくいので状態作りには使わない。
- 演出の弱強・種別はデバッグパネルの「予告演出:」行で裏取りできる(見た目だけで判断しない)。
- **液晶表示の細部確認は遊技ボタン列の「液晶2倍」トグル**(#097 新設。液晶を実寸 2 倍の固定パネルへ複製表示。SE は本体側のみ再生)。
- バトルなど深いフェーズの状態作りは、`App.tsx` の `initGameState` 直後に一時的に `initialState.phase` を固定するのが早い(例: `{ type: 'AT', tier: 'UPPER', part: 'KOYAKU', partGame: 10, … }` = 次レバーでバトル 1G 目。**continueRate はテーブルの実在値(下位 = 0.66/0.79/0.84/0.88・上位 = 0.93)にしないと `drawBattleRoute` が throw する**。確認後は必ず revert)。
- 一時的なテーブル固定(演出を 100% 出す等)で確認した場合は**必ず revert してからコミット**する。
- VM に音声デバイスはない(実音の確認は Windows 実機確認に委ねる)。
- シミュレーションは `npm run sim -- <ゲーム数> <シード>`(100 万 G 約 13 秒)。出玉レポートは `docs/SIMULATION_REPORT.md`。
